[~] TENANT-094 — Gestione completa utenti organizzazione e controllo self-signup
------------------------------------------------------------------------------
  Priorita: ALTA
  Dipendenze: TENANT-021 (completato)

  DESCRIZIONE:
  Permettere a OWNER e ADMIN di gestire completamente gli utenti della propria
  organizzazione dalla WebUI: creazione diretta di nuovi account (non solo invito
  di utenti esistenti), abilitazione/disabilitazione utenti, e modifica ruoli.
  Aggiungere un toggle per disabilitare/abilitare la self-registration sia come
  variabile d'ambiente (SELF_SIGNUP_ENABLED) sia come impostazione runtime nella
  tab Administration. Se la self-registration e' disattivata, solo OWNER/ADMIN
  possono creare nuovi utenti. Se il servizio email e' configurato, inviare una
  notifica all'utente con le credenziali di accesso.

  DETTAGLI TECNICI:

  1. Schema Prisma (server/prisma/schema.prisma):
     - Nuovo modello AppConfig (key-value store per configurazione globale):
         model AppConfig { key String @id; value String; updatedAt DateTime @updatedAt }
     - Nuovo campo User.enabled (Boolean @default(true)) per abilitare/disabilitare utenti
     - Nuove AuditAction: TENANT_CREATE_USER, TENANT_TOGGLE_USER, APP_CONFIG_UPDATE
     - Generare migrazione Prisma

  2. Configurazione server (server/src/config.ts + .env.example):
     - Aggiungere selfSignupEnabled: process.env.SELF_SIGNUP_ENABLED !== 'false'
     - Aggiungere SELF_SIGNUP_ENABLED=true in .env.example

  3. Nuovo servizio AppConfig (server/src/services/appConfig.service.ts):
     - getSelfSignupEnabled(): legge da DB (tabella AppConfig, chiave selfSignupEnabled),
       fallback su config.selfSignupEnabled (env var) se non presente in DB
     - setSelfSignupEnabled(enabled): upsert chiave selfSignupEnabled in AppConfig
     - getPublicConfig(): restituisce { selfSignupEnabled } per client non autenticati
     - Cache in-memory con TTL breve per evitare query ad ogni registrazione

  4. Guard self-signup su registrazione (server/src/services/auth.service.ts):
     - In register(): verificare getSelfSignupEnabled(), lanciare AppError 403 se disattivato
     - In login(): verificare user.enabled, bloccare con 403 se account disabilitato
     - In refreshAccessToken(): verificare user.enabled, invalidare token se disabilitato
     - Verificare anche nel flusso OAuth callback per bloccare registrazioni OAuth

  5. Creazione utente da admin (server/src/services/tenant.service.ts):
     - Nuova funzione createUser(tenantId, { email, password, role, username }, actingUserId)
     - Verifica email non duplicata (distinguere: gia' nel tenant / in altro tenant / senza tenant)
     - Setup vault encryption (Argon2 salt, master key, derived key, AES-256-GCM) identico a register()
     - emailVerified = true (vouched by admin), assegnazione diretta a tenant con ruolo
     - Nuova funzione toggleUserEnabled(tenantId, targetUserId, enabled, actingUserId)
     - Protezioni: non disabilitare se stessi, non disabilitare ultimo OWNER

  6. Controller e route (server/src/controllers/tenant.controller.ts, tenant.routes.ts):
     - POST /:id/users → createUser (requireTenantRole('ADMIN'))
     - PATCH /:id/users/:userId/enabled → toggleUserEnabled (requireTenantRole('ADMIN'))
     - Validazione con Zod (email, password min 8, role ADMIN|MEMBER, sendWelcomeEmail boolean)
     - Audit logging per ogni azione

  7. Endpoint self-signup toggle (server/src/routes/admin.routes.ts, admin.controller.ts):
     - GET /admin/app-config → stato attuale { selfSignupEnabled }
     - PUT /admin/app-config/self-signup → toggle { enabled: boolean }
     - GET /auth/config → endpoint pubblico (non autenticato) per il client

  8. Email di benvenuto (server/src/services/email/index.ts):
     - Nuova funzione sendWelcomeEmail(to, temporaryPassword)
     - Template HTML/text con: login URL, email, password temporanea
     - Dev mode: log su console come sendVerificationEmail

  9. API client frontend (client/src/api/tenant.api.ts, admin.api.ts, auth.api.ts):
     - createTenantUser(tenantId, data) → POST /tenants/:id/users
     - toggleUserEnabled(tenantId, userId, enabled) → PATCH /tenants/:id/users/:userId/enabled
     - getAppConfig() e setSelfSignup(enabled) in admin.api.ts
     - getPublicConfig() in auth.api.ts

  10. Store frontend (client/src/store/tenantStore.ts):
      - Nuove azioni: createUser(), toggleUserEnabled()
      - Aggiornare tipo TenantUser con campo enabled

  11. UI — CreateUserDialog (client/src/components/Dialogs/CreateUserDialog.tsx):
      - Campi: email (required), username (opzionale), password (min 8), conferma password,
        ruolo (ADMIN|MEMBER), checkbox "Invia email di benvenuto" (visibile solo se email configurata)
      - Bottone genera password casuale (16 caratteri)
      - Mostra riepilogo credenziali dopo creazione

  12. UI — TenantSection aggiornata (client/src/components/Settings/TenantSection.tsx):
      - Aggiungere bottone "Create User" accanto a "Invite"
      - Nuova colonna "Status" nella tabella utenti con Switch enable/disable
      - Protezione: non mostrare switch per il proprio account

  13. UI — SelfSignupSection (client/src/components/Settings/SelfSignupSection.tsx):
      - Card con Switch "Allow new users to register themselves"
      - Helper text: "When disabled, only organization admins can create user accounts"
      - Aggiungere in SettingsDialog.tsx nella tab Administration

  14. UI — RegisterPage (client/src/pages/RegisterPage.tsx):
      - On mount: chiamare getPublicConfig()
      - Se selfSignupEnabled === false: mostrare messaggio info e disabilitare form
